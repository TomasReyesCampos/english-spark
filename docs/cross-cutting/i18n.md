> Estrategia de localización: idiomas soportados, tono, formato de fechas
> y números, currency conversion, RTL (no aplicable), pricing localizado,
> contenido culturalmente apropiado por país.

**Owner:** —

## 1. Scope

### 1.1 Lo que se localiza

- **Idioma de UI:** español (variantes mínimas, ver §2).
- **Pricing:** moneda local por país.
- **Formato de fechas y números:** según locale.
- **Contenido cultural específico:** algunos ejemplos en roleplays,
  logros culturales (`cinco_de_mayo`, `dia_del_amigo`, etc.).
- **Eventos calendario:** Día del Estudiante, vacaciones por país.
- **Currency en notificaciones de pago:** según moneda de la
  transacción.

### 1.2 Lo que NO se localiza

- **Inglés del producto:** se enseña inglés americano, británico o
  neutral según `target_english_variant` del usuario (ver
  `pedagogical-system.md` §3.6). El inglés target es el contenido
  pedagógico, no UI.
- **Logs internos, eventos, errors técnicos:** todo en inglés.
- **Documentación técnica del repo:** español neutro (decisión del
  owner; podría cambiar si crece el equipo).

### 1.3 RTL

No aplicable. No soportamos idiomas RTL en MVP ni post-MVP previsto.

## 2. Idioma de interfaz: español neutro latinoamericano

### 2.1 Variante única en MVP

**Decisión:** un solo locale `es` con redacción neutra latinoamericana.
NO soportamos en MVP variantes separadas (`es-MX`, `es-AR`, `es-CO`).

**Razón:**
- Mantener **un solo set de strings** reduce complejidad y errores de
  traducción.
- Diferencias entre variantes son menores en escritura formal/de
  producto.
- Cuando hay variación significativa, se elige la opción más universal.

### 2.2 Reglas de tono y vocabulario

#### Sí

- Voseo / tuteo: usar **vos** o **tú** consistentemente. Decisión por
  default: **vos** (más rioplatense pero entendible en toda Latam).
  - Si métricas muestran rechazo en México/Colombia, cambiar a **tú**
    en MVP+1.
- Tono cercano pero profesional. Tratar al usuario como un par adulto.
- "Te acompañamos en tu camino", no "¡Vamos campeón!".

#### No

| Evitar | Por qué |
|--------|---------|
| Modismos muy locales: "chido", "guay", "bárbaro", "bacán" | Excluyen otros países |
| Diminutivos excesivos: "ratito", "cuentita" | Suena infantil |
| Inglés innecesario: "click", "deadline", "sprint" en UI usuario | Hay equivalente español natural |
| "Tú lo lograste, campeón" / "¡Eres un crack!" | Tono gringo traducido |
| Albur o doble sentido | Mercado profesional |
| Palabras religiosas casuales: "santo", "milagro" | Excluye sensibilidades |

#### Tabla de decisiones específicas

| Concepto | Forma elegida | Forma rechazada |
|----------|--------------|-----------------|
| Pronombre 2da persona | "vos" | "tú" (alternativa válida) |
| "you can" imperativo | "podés" o "puedes" (consistente) | "tu puedes" |
| "Sign up" | "Registrate" | "Inscríbete", "Crea tu cuenta" |
| "Track" (camino de aprendizaje) | "track" (mantener inglés, es término del producto) | "ruta", "camino" |
| "Streak" | "racha" | "streak" |
| "Sparks" | "Sparks" (marca, no traducir) | "chispas" |
| "Mastery" | "dominio" | "maestría" (suena académico) |
| "Block" | "bloque" | "lección" (sería confuso con asset) |
| "Daily goal" | "objetivo del día" | "meta diaria" |
| "Free trial" | "prueba gratis" | "trial gratuito" |
| "Achievements" | "logros" | "trofeos" |
| "Streak freeze" | "salvador de racha" | "congelador de racha" |
| Fechas | "el lunes 5 de mayo" | "lunes, mayo 5" |
| Moneda | "$100 MXN" o "100 pesos mexicanos" según contexto | "$100 mexican pesos" |

### 2.3 Mensajes con datos dinámicos

Ejemplo: "Llevás 8 días seguidos practicando".

Convención de placeholders:

```typescript
{
  "streak.current": "Llevás {days} días seguidos practicando",
  "sparks.balance": "Te quedan {sparks} Sparks",
  "block.completed": "Completaste {blockName}",
}
```

Pluralización:

```typescript
{
  "streak.current_one": "Llevás {days} día seguido practicando",
  "streak.current_other": "Llevás {days} días seguidos practicando",
}
```

Usar librería `i18next` o `react-intl` que soporte ICU MessageFormat
para pluralización en español (un, varios).

### 2.4 Tono específico por contexto

| Contexto | Tono |
|----------|------|
| Onboarding | Cálido, motivador, expectativa positiva |
| Ejercicio en curso | Conciso, claro, sin distracción |
| Feedback de error pedagógico | Constructivo, sin culpa: "Estuviste cerca. Probá..." |
| Logro desbloqueado | Celebratorio sin exageración |
| Streak en peligro | Urgente pero amigable |
| Paywall | Honesto, valor clarísimo, sin presión |
| Anti-fraud restriction | Formal, neutro, claro sobre apelación |
| Soporte (AI Assistant) | Empático, soluciones-first |
| Notificación de pago fallido | Práctico, instructivo |

## 3. Formato

### 3.1 Fechas

| Locale | Formato corto | Formato largo |
|--------|--------------|---------------|
| `es` (default) | `5/5/2026` | `5 de mayo de 2026` |
| `es-AR` (override) | `5/5/2026` | `5 de mayo de 2026` |
| `es-MX` (override) | `5/5/2026` | `5 de mayo de 2026` |

Implementación: `Intl.DateTimeFormat('es', options)`. Locale del navegador
o explícito según preferencia del usuario.

### 3.2 Números

```typescript
new Intl.NumberFormat('es', { useGrouping: true }).format(1234567);
// "1.234.567" (España) o "1,234,567" (México) — depende del subtag
```

Decisión: **usar separador de miles `.` y decimales `,`** (norma
latinoamericana mayoritaria). Ejemplo: "1.234,56".

### 3.3 Moneda

```typescript
new Intl.NumberFormat('es-MX', { style: 'currency', currency: 'MXN' }).format(100);
// "$100.00"
```

Cada país → moneda preferida (§4). Mostrar símbolo + código cuando
hay ambigüedad (ej: "$100 MXN" en country selector si pueden ver
varios precios).

### 3.4 Tiempo y timezone

- **Timezone:** detectado del device, override editable en settings.
- **24h o 12h:** según `Intl.DateTimeFormat` por locale (default).
- **Persistencia interna:** SIEMPRE UTC + ISO-8601 con timezone offset.

### 3.5 Pluralización

Usar plural rules de español:
- **one:** 1 (ej: "1 día")
- **other:** 0, 2, 3, ... (ej: "0 días", "2 días")

NO existe `few` o `many` en español; no usar.

## 4. Pricing por país

### 4.1 Tabla de pricing localizado

Pricing **fijo** en moneda local (no recalculado en runtime salvo
Argentina). Revisar trimestralmente.

| País | Moneda | Plan Básico | Plan Pro | Plan Premium | Pack Pequeño | Pack Mediano | Pack Grande |
|------|--------|------------:|---------:|-------------:|-------------:|-------------:|------------:|
| México | MXN | $30 | $100 | $250 | $50 | $130 | $350 |
| Argentina | USD (con conversión runtime) | $1.50 | $5 | $13 | $2.50 | $7 | $18 |
| Colombia | COP | $7.000 | $22.000 | $55.000 | $11.000 | $29.000 | $77.000 |
| Chile | CLP | $1.500 | $5.000 | $12.500 | $2.500 | $6.500 | $17.500 |
| Perú | PEN | $7 | $22 | $55 | $11 | $29 | $77 |
| Otros Latam | USD | $1.50 | $5 | $13 | $2.50 | $7 | $18 |
| España | EUR (post-MVP) | TBD | TBD | TBD | TBD | TBD | TBD |
| EEUU hispano | USD | $2 | $7 | $18 | $4 | $10 | $25 |

### 4.2 Argentina: pricing dinámico

Argentina tiene inflación significativa. Estrategia:

- Pricing en **USD** internamente.
- Conversión a ARS al momento del cobro usando tipo de cambio
  oficial (Banco Nación) + spread del 10% (cubre fluctuación entre
  cobro y settlement).
- UI muestra `$X USD (≈ Y ARS)` para evitar sorpresas.
- Revisión mensual del spread.

**No** ajustamos precios USD para Argentina. Si el USD sube en Latam,
nuestro pricing sigue siendo estable en USD para todos.

### 4.3 In-app purchases (stores)

Apple App Store y Google Play tienen **tiers de pricing** que no
podemos personalizar fila por fila. Mapeo de tiers:

| Plan | Apple Tier | Google Tier (price points) |
|------|-----------|---------------------------|
| Básico mensual | Tier 1 (~$1.99 USD) | $1.99 |
| Pro mensual | Tier 5 (~$4.99 USD) | $4.99 |
| Premium mensual | Tier 13 (~$12.99 USD) | $12.99 |
| Pack Pequeño | Tier 2 (~$2.99 USD) | $2.99 |
| Pack Mediano | Tier 7 (~$6.99 USD) | $6.99 |
| Pack Grande | Tier 18 (~$17.99 USD) | $17.99 |

Cada tier se configura por país en App Store Connect / Google Play
Console con los precios listados arriba.

### 4.4 Web pricing (Stripe / MercadoPago)

Web tiene **pricing dinámico** sin restricción de tiers.

Estrategia: **mostrar 15% de descuento web** vs in-app:

```
Plan Pro:
  $5 USD/mes (in-app, vía App Store o Play Store)
  $4.25 USD/mes (en web, pagando con tarjeta o MercadoPago)

¡Ahorrá 15% pagando desde web!
```

Esto es **legalmente permitido** en muchas jurisdicciones (verificar
por país antes de implementar; cambia con frecuencia).

### 4.5 Detección de país para pricing

Orden de prioridad:

1. País en `student_profiles.country` (declarado por el usuario).
2. País detectado vía IP (Cloudflare provee `cf.country`).
3. Default: USD pricing genérico.

**Importante:** mostrar siempre el país detectado en la UI con opción
de cambiar. Evita confusión cuando alguien viaja.

### 5.5 Notificaciones contextualmente apropiadas

Push notifications consideran el contexto:

```typescript
// Ejemplo: notificación de daily reminder
function getReminderMessage(user: User): { title: string; body: string } {
  const hour = getUserLocalHour(user);
  const isWeekend = isWeekend(user);

  if (hour < 12) {
    return { title: `¡Buen día, ${user.name}!`, body: '...' };
  }
  if (hour >= 18) {
    return { title: `Cerrá el día con inglés, ${user.name}`, body: '...' };
  }
  return { title: `Hola, ${user.name}`, body: '...' };
}
```

NO traducir literalmente "Good morning" → "Buenos días" en una app que
notifica a las 19hs.

## 6. Detección de país y timezone

### 6.1 Onboarding

Pregunta directa al usuario en onboarding:

```
¿Dónde estás?
[🇲🇽 México] [🇦🇷 Argentina] [🇨🇴 Colombia]
[🇨🇱 Chile] [🇵🇪 Perú] [🌎 Otro]
```

Pre-selección basada en IP (Cloudflare) pero el usuario confirma.

### 6.2 Cambio posterior

Settings permite cambiar país. Cambia:
- Pricing mostrado.
- Logros culturales disponibles.
- Eventos calendario.
- Default `target_english_variant` (sugerencia, editable).

Cambio de país de un usuario activo no recalcula su roadmap automáticamente.

### 6.3 Timezone

- **Detección:** `Intl.DateTimeFormat().resolvedOptions().timeZone` en
  cliente. Persistido en `user_notification_preferences.timezone`.
- **Override:** editable en settings.
- **Cambios automáticos por viaje:** NO — el usuario que viaja a otro
  TZ debe cambiar manualmente si quiere que las notifs respeten el
  nuevo TZ.

## 7. Stack técnico

### 7.1 Cliente (React Native + Web)

- **Librería:** `react-i18next` (cobertura completa de pluralización
  ICU, lazy loading de bundles, namespaces).
- **Storage de strings:** archivos JSON por locale en
  `apps/mobile/locales/<locale>.json`.
- **Detección inicial:** `expo-localization` da locale del device.
- **Fallback:** si una key no existe en `es`, fallback a la key (no a
  inglés). Prefer estricto: cualquier missing key es bug.

### 7.2 Backend (Workers)

Para mensajes que el backend genera (ej: AI Assistant prompts,
notification bodies generados por IA, emails):

- Mensajes en español, hardcoded en código de Workers.
- Ningún string del backend es localizable a otro idioma en MVP (todo
  es español).
- Cuando AI Gateway genera contenido (notifs, insights), el prompt
  especifica español neutro latinoamericano.

### 7.3 Stores (App Store + Play Store)

- Páginas localizadas: español (es), español-LATAM (es-419), inglés (en) post-MVP.
- Capturas localizadas: español por país clave (MX, AR, CO).
- Keywords ASO: específicas por país.

### 7.4 Email transaccional

Stripe / RevenueCat envían emails de receipts en idioma del usuario
configurado en su cuenta de pago. NO controlamos esto.

Emails que enviamos nosotros (welcome, account deletion confirmation)
están en español neutro.

### 7.5 Estructura de archivos de traducción

```
apps/mobile/locales/
├── es.json                  ← strings principales
├── es-AR.json               ← override mínimo (post-MVP)
├── es-MX.json               ← override mínimo (post-MVP)
└── pt-BR.json               ← post-MVP

# Estructura interna por archivo
{
  "common": {
    "ok": "OK",
    "cancel": "Cancelar"
  },
  "onboarding": {
    "welcome.title": "...",
    "welcome.body": "..."
  },
  "sparks": {
    "balance.label": "Tu balance",
    "low.title_one": "Te queda {sparks} Spark",
    "low.title_other": "Te quedan {sparks} Sparks"
  },
  ...
}
```

## 8. Post-MVP

### 8.1 Portugués brasilero (pt-BR)

Cuando se valide el producto en Latam, expandir a Brasil. Implica:

- Nuevo locale `pt-BR` con traducción profesional (no hispanoparlante
  improvisada).
- Pricing en BRL.
- PIX como método de pago.
- Variante de inglés default americana.
- Logros culturales brasileros.
- Track de "Inglês para o trabalho" adaptado.

### 8.2 Inglés UK / hispanos en EEUU

Comunidad hispana en EEUU es un mercado significativo. Implica:

- Pricing en USD.
- Considerar inglés como UI alternativa (algunos usuarios prefieren
  UI en inglés).
- Métodos de pago US (tarjetas, Apple/Google Pay).

### 8.3 España

Mercado más pequeño que Latam pero significativo. Implica:

- Locale `es-ES` con tuteo (España no usa voseo).
- Vocabulario y modismos diferentes ("vale", "ordenador", "móvil").
- Pricing en EUR.
- Cumplimiento estricto de GDPR.

### 8.4 Variantes españolas (post-MVP)

Si métricas muestran que voseo afecta conversion en MX/CO:

- Override `es-MX` con tuteo.
- Override `es-AR` con voseo.
- Override `es-CO` con tuteo + algunas frases locales.

Cada override es un archivo JSON con solo las keys que difieren del
default.
